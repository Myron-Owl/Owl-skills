# 1st-pr — 第一性原理思维

拆解问题至基本真理，从底层重新构建解决方案。与"类比思维"（reasoning by analogy）相对。

## 这个 Skill 是做什么的？

当你面对复杂问题、传统方法解决不了、或者感觉现有方案只是"大家都这么做"时，这个 skill 引导 Claude 使用**第一性原理**来拆解和分析问题——不是直接跳到答案，而是回到最基本的物理/数学/经济事实，再重新往上构建。

## 什么时候用

- 传统方法解决不了的复杂问题
- 需要质疑现有假设时
- 反复出现的 Bug 需要根因分析
- 架构设计和技术选型
- 听到"我们一直这么做"、"行业标准是xxx"时

## 使用方式

在 Claude Code 中：

```
/1st-pr 为什么电动车比燃油车加速更快？
/1st-pr 这个系统的架构是否合理？
/1st-pr 我们应该用微服务吗？
```

也支持自然语言触发，比如"用第一性原理思考：房价为什么一直涨？"

## 五步流程

### Step 1：明确问题
清晰界定问题边界、范围、成功标准。

### Step 2：拆解至基本元素
把问题拆到不可再分的基本真理，区分"事实"和"假设"。

拆解维度：
| 维度 | 追问 |
|------|------|
| 物理/硬约束 | 什么是不可改变的事实？ |
| 逻辑/数学 | 什么是数学上必然成立的？ |
| 领域定律 | 这个领域的公理是什么？ |
| 目标/需求 | 用户真正需要的是什么？ |

### Step 3：挑战假设
识别"我们一直这么做"的隐含假设，逐条标记为已验证事实、未经检验的假设、或可能过时的假设。

### Step 4：从零重建
基于基本元素，不受现有方案影响，重新设计解决方案。至少产出 3 个方向完全不同的方案。

### Step 5：验证计划
制定最小验证实验，主动证伪而非证明。

## 内置应用框架

| 框架 | 适用场景 |
|------|---------|
| 软件架构拆解 | 系统设计、性能优化、技术选型 |
| Bug 根因拆解 | 复杂 Bug 排查、线上事故分析 |
| 产品/功能设计拆解 | 需求分析、UX 改进 |

## 输出模板

每个分析输出遵循标准格式，包含：问题定义 → 基本元素 → 假设挑战 → 从零重建 → 验证计划 → 结论。

## 提防的认知偏差

- **确认偏误**：只找支持自己观点的证据 → 主动寻找证伪证据
- **沉没成本谬误**："已经投入这么多不能改" → 忽略过去投入，只看未来
- **权威偏误**："某公司也这么做" → 权威做法 ≠ 最佳做法
- **框架效应**：问题呈现方式影响解决方向 → 用多种方式重新描述问题

## 核心原则

1. 区分事实与假设 — 核心动作
2. 质疑一切，但有方法
3. 从物理/数学上确认什么是不可能的 — 剩下的都是可能的
4. 拒绝"行业标准答案"
5. 推理链必须透明，每一步都能检验
